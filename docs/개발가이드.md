# 개발 가이드

## 이 프로젝트는 최소 뼈대입니다

패키지 구조, 엔티티 필드, DTO 필드, 클래스 배치만 잡혀 있습니다.
검증 애노테이션, Controller 매핑, Service/Repository 로직은 **각자 담당 TODO를 보고 직접 구현합니다.**

## 미리 확정된 것 (변경 시 팀 협의 필수)

- **엔티티 필드와 연관관계** — ERD 기준. 다른 도메인이 참조하므로 마음대로 바꾸지 않기
- **DTO 필드 구성** — API 명세서 기준 (docs/API_명세서.md)
- **URL 설계** — 각 Controller의 TODO 주석에 명세 있음
- **Product의 재고 메서드 이름** — `decreaseStock(int)` / `increaseStock(int)` (5번·6번 공용)
- **공통 예외** — common.exception의 예외를 던지면 GlobalExceptionHandler(구현 필요)가 상태코드로 변환
  - IllegalArgumentException→400 / Unauthorized→401 / Forbidden→403 / NotFound→404 / Duplicate→409

## 담당별 작업 범위

| 번호 | 담당 | 구현할 것 |
|------|------|----------|
| 1번 | 관리자 회원가입 | Admin 생성 메서드, SignupRequest 검증, AdminService.signup, Controller 매핑 |
| 2번 | 관리자 인증 | AuthService.login, AuthController(로그인/로그아웃), 세션 저장 |
| 3번 | 관리자 정보 관리 | Admin 수정/승인/거부 메서드, AdminService·Controller 나머지 전체 |
| 4번 | 고객 정보 관리 | customer 패키지 전체 |
| 5번 | 상품 정보 관리 | product 패키지 전체 + 재고 메서드(decreaseStock/increaseStock) |
| 6번 | 주문 정보 관리 | order 패키지 전체 |

- GlobalExceptionHandler는 팀 협의로 담당자 1명 지정 (또는 가장 먼저 필요한 사람이 구현 후 공유)

## 공통 규칙

- 엔티티 setter 금지 — 의미 있는 비즈니스 메서드로 상태 변경
- 페이징 응답은 common.dto.PageInfo 사용
- 다른 사람 도메인 코드 수정 금지 (필요 시 협의)

## 브랜치 규칙

main에서 자기 브랜치 생성 → 작업 → PR → 리뷰 → main 머지
머지 후 전원: `git checkout main && git pull && git checkout 자기브랜치 && git merge main`

## 실행

```bash
./gradlew bootRun
# H2 콘솔: http://localhost:8080/h2-console (JDBC URL: jdbc:h2:mem:commercedb)
```
